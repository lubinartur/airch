# Spaces — Operational Spec

> How AIRCH maintains and presents domain rooms.

---

## Role

A Space is a room AIRCH has been keeping for a domain of life.

It contains memory, open questions, momentum, and AIRCH's current read on this part of the user's life.

Entering a Space should feel like walking into a prepared room — not like opening a folder.

---

## Space Types

### System Spaces
Created automatically when sufficient domain data exists:
- Finance
- Health
- Projects (or Work / Startup — named by the user)

System Spaces cannot be deleted, only archived if dormant.

### User Spaces
Created by the user or proposed by AIRCH when a domain reaches sufficient weight in memory.

Examples: Learning, Ducati, Family, Side project, Travel

### AIRCH-proposed Spaces
When AIRCH detects that a topic has appeared in Dialogue or domain data with sufficient frequency and significance, it may propose a new Space:

> "You have mentioned [topic] several times over the past month. Do you want me to keep a room for it?"

Threshold for proposal: topic appears in ≥ 5 distinct interactions within 30 days, across ≥ 2 different contexts.

The user can accept, decline, or say "not yet."

---

## Space Content

Each Space contains:

### AIRCH's Read (required)
One sentence. AIRCH's current assessment of this domain.

Generated by the Reasoning Engine using domain-specific Memory and recent signals. Updated when new relevant data arrives or when significant time has passed (14 days without update → reassessed).

Examples:
> Three things are active. One is moving. Two have not been touched in eleven days.
> Spending is stable. One subscription has not been used in 47 days.
> Recovery has been declining for two weeks. The pattern precedes your worst training periods.

If nothing meaningful has changed and AIRCH has no insight: the previous Read remains. AIRCH does not manufacture a new one.

### Open Questions (optional)
Unresolved topics specific to this Space. Sourced from:
- Open loops from Dialogue that belong to this domain
- Decisions pending in this domain
- Patterns that need user input to resolve

Maximum 3 shown. The rest are held until relevant.

### Momentum Signal (required)
A directional indicator: is this domain moving, stalled, or steady?

Not a progress bar. Not a percentage. A qualitative signal:
- Moving — recent activity, things changing
- Steady — consistent, no major shifts
- Stalling — activity declining, nothing new
- Dormant — no meaningful activity for 30+ days

Determined by: recency of meaningful events, rate of change in domain data, user engagement with this Space.

### Recent Events (optional)
The last 3–5 meaningful events in this domain. One sentence each. Date shown.

Not a feed. Not a log. The events that matter for understanding current state.

### Connected Threads (optional)
Threads that touch this Space. Listed by name only. The user taps to explore.

Shown only if active (a Thread that has closed is not shown here).

---

## Space Updates

### When a Space updates
- New domain data arrives (finance upload, health sync)
- A relevant Dialogue session ends (extraction produces new Facts or Events)
- A significant amount of time passes without update (14 days → reassessment)
- User explicitly interacts with the Space

### What triggers reassessment of AIRCH's Read
- New high-confidence signal in the domain
- A Pattern has changed status (confirmed, dormant, reactivated)
- User takes a significant action (archives a project, closes a subscription)
- 14 days without update

### What does NOT trigger a Space update
- Low-confidence observations
- Routine domain data with no anomaly (transactions within normal range, workout within normal pattern)
- External noise (market data, news) without personal relevance

---

## Space Entry

From navigation or Today: Space Card.

Space Card shows:
- Space name
- AIRCH's Read (one sentence)
- Momentum signal (quiet visual indicator)
- Last meaningful activity (date)

Nothing more. Depth is inside.

---

## Space Dormancy

A Space becomes dormant when:
- No meaningful activity for 60 days
- Momentum signal is "Dormant" for 30+ consecutive days

AIRCH may note this in Dialogue: "The [Space] room has been quiet for two months. Is it still relevant?"

The user can:
- Keep it active (AIRCH continues monitoring)
- Archive it (Space hidden from navigation; memory retained)
- Delete it (Space and associated memory removed)

A dormant Space is not automatically archived. AIRCH asks.

---

## Space-Specific Agent

Each Space has a domain-specific reasoning context. When the user is in a Space, the Reasoning Engine applies domain-specific logic and tone.

Finance Space: analytical, numbers-focused, no emotion
Health Space: direct, pattern-focused, no lecturing
Projects Space: momentum-focused, honest about stalls

This is not a different AIRCH — it is the same presence in a different room, with relevant expertise.

---

## Dialogue Within a Space

The user can open Dialogue from within a Space. When they do:
- Dialogue context is loaded with the Space's memory
- AIRCH responds as if focused on this domain
- New information extracted goes back into the Space's memory

The user can also exit the Space context during conversation — AIRCH follows the thread wherever it leads.

---

## Edge Cases

**Two Spaces with overlapping domain:**
E.g. "Work" and "AIRCH" (the startup). Both exist. They share some memory but have distinct focuses. Threads may connect them.

**Space proposed but user declines:**
AIRCH does not propose the same Space again for 60 days. After 60 days, if the threshold is met again, AIRCH proposes once more.

**User renames a Space:**
The name changes. The memory and structure remain.

**Conflicting signals within a Space:**
E.g. Finance shows normal spending but a specific category is spiking. AIRCH's Read reflects the most significant signal, not an average. The spike is the story.

**User asks AIRCH to "reset" a Space:**
AIRCH clarifies what this means — clear the history, start fresh observations, or just archive old content. It does not silently wipe memory based on an ambiguous request.
